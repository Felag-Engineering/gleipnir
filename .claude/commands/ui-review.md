# UI Review

You are a senior UI/UX designer and frontend engineer reviewing the Gleipnir interface. Work through the stages below in order. Be thorough but concise — your final review should be specific and actionable.

## Arguments

`$ARGUMENTS` — check for the `--post-to-pr` flag.

## Design system reference

Before reviewing, internalise this spec:

- **Theme:** dark. Canvas `#0F1117`, surface `#131720`, elevated `#1E2330`, top bar `#0D1018`.
- **Text:** faint `#334155`, muted `#475569`, secondary `#94A3B8`, primary `#E2E8F0`.
- **Accent colours:** blue `#60A5FA` (sensors, running), orange `#FB923C` (actuators), amber `#F59E0B` (approvals), green `#4ADE80` (success/complete), red `#F87171` (errors/failed), purple `#A78BFA` (feedback/interrupted).
- **Typography:** IBM Plex Sans body, IBM Plex Mono for code/monospace. Scale: 11/13/15/18/24/32px.
- **Spacing:** 4px base grid — all margins, padding, gaps snap to 4, 8, 12, 16, 24, 32, 48, 64px.
- **Borders:** subtle `#1E2330`, mid `#253044`.

## Stage 1 — Start the environment

Run the setup script. If it fails, diagnose the error and fix it before continuing.

```bash
bash scripts/start-review-env.sh
```

Confirm both services are up:
- App: `http://localhost:3000`
- Storybook: `http://localhost:6006`

---

## Stage 2 — App page screenshots

Set the browser viewport to 1440×900, then visit each route. For each page:
1. Navigate and wait for the page to settle (use `browser_wait_for` with a short selector poll if needed).
2. Take a full-page screenshot.
3. Take an accessibility snapshot for structural analysis.
4. Note any issues immediately as you see them.

Pages to review:

| Route | Label |
|---|---|
| `http://localhost:3000/dashboard` | Dashboard |
| `http://localhost:3000/policies/new` | Policy Editor — new |
| `http://localhost:3000/mcp` | MCP Registry |

Also try a 375×812 viewport (mobile) on the Dashboard to check responsive behaviour.

---

## Stage 3 — Storybook story screenshots

Fetch the story index from Storybook's API:

```
http://localhost:6006/index.json
```

Use `browser_evaluate` to parse the JSON and get the full list of story IDs. Then for each story:

1. Navigate to `http://localhost:6006/iframe.html?id=<storyId>&viewMode=story`
2. Wait 600ms for animations to settle.
3. Take a screenshot.

Visit **every story** — do not sample. The story count is small enough to cover completely.

---

## Stage 4 — Structured review

Write a structured review covering:

### 1. Critical issues
Layout breakages, contrast failures (WCAG AA minimum 4.5:1 for text), broken states, misaligned elements.

### 2. Consistency gaps
Spacing that doesn't snap to the 4px grid, colour usage that deviates from the palette, typography that mismatches the scale, border colours out of spec.

### 3. UX improvements
Flows that are confusing, missing affordances, unclear empty states, unhelpful error messages.

### 4. Component-by-component notes
For each Storybook story group, one or two sentences on visual polish and story coverage (e.g. "missing loading/error story").

### 5. Positive observations
What is working well — anchor future work to these strengths.

Format the entire review as **GitHub-flavoured markdown**. Reference specific component names, route paths, and CSS custom property names where relevant. Prioritise issues by impact (critical → minor).

---

## Stage 5 — Post to PR (conditional)

If `--post-to-pr` was passed:

1. Find the current open PR for this branch:
   ```bash
   gh pr view --json number,title,url
   ```
2. Post the review as a PR comment:
   ```bash
   gh pr comment <number> --body "$(cat <<'REVIEW_EOF'
   ## 🔍 UI Review

   <your review here>

   ---
   _Generated by `/ui-review` — live Playwright screenshots of the running stack_
   REVIEW_EOF
   )"
   ```
3. Print the PR URL so the user can navigate directly to it.

If `--post-to-pr` was **not** passed, output the full review directly to the terminal.

---

## Rules

1. **Do not skip stages.** Run the env setup even if you think the stack is already up — the script is idempotent.
2. **Screenshot every story.** Do not stop at a sample.
3. **Be specific.** Vague feedback like "needs improvement" is not useful. Name the component, describe the exact problem, suggest a concrete fix.
4. **Stop on environment failures.** If Docker Compose won't start or Storybook fails to build, report the error clearly and stop.
